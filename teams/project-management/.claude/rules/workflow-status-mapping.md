# Workflow Status Mapping

## 規則內容

### 需求工作流（Requirement Workflow）

需求工單的狀態流轉順序為：

```
草稿 → 需求評審 → 待規劃 → 規劃中 → 待設計 → 設計中 → 待排期 → 待開發 → 開發中 → 待測試 → 測試中 → 已關閉
```

你必須依照上述順序進行狀態追蹤與指標計算。不得自行新增、刪除或調換狀態。

### 開發工作流（Development Workflow）

開發工單的狀態流轉順序為：

```
待排期 → 待執行 → 執行中 → 驗收中 → 已關閉
```

你必須依照上述順序進行狀態追蹤與指標計算。不得自行新增、刪除或調換狀態。

### 狀態分類（Status Classification）

所有狀態分為三類：

**等待中狀態（Waiting）**：以「待」開頭的狀態，表示工單正在等待下一階段的處理。

- 需求：待規劃、待設計、待排期、待開發、待測試
- 開發：待排期、待執行

**活躍中狀態（Active）**：以「中」結尾的狀態或特殊階段，表示工單正在被處理。

- 需求：草稿、需求評審、規劃中、設計中、開發中、測試中
- 開發：執行中、驗收中

**已關閉狀態（Closed）**：

- 需求：已關閉
- 開發：已關閉

### 分類用途

- 停滯計算僅適用於「等待中（Waiting）」狀態，不適用於「活躍中（Active）」或「已關閉（Closed）」狀態。
- 週期時間（Cycle Time）計算適用於所有狀態。
- 完成率計算以「已關閉（Closed）」狀態為分子。

## 違規判定（Violation Determination）

- 在計算中使用不存在於上述定義中的狀態名稱 → 違規
- 將「活躍中」狀態納入停滯計算 → 違規
- 調換狀態流轉順序（例如將「待設計」放在「規劃中」之前） → 違規
- 將需求工作流的狀態套用到開發工單 → 違規

### 違規情境

計算停滯指標時，將一張處於「開發中」狀態超過 14 天的 Highest 需求工單標記為停滯。此判定違規，因為「開發中」屬於活躍中（Active）狀態，停滯計算僅適用於等待中（Waiting）狀態。
