# Priority Ranking

## 目的

為 RDC 專案（TutorABC Education）的工單提供標準化排序邏輯，並更新 JIRA 的 Rank Number 欄位（`customfield_10237`）。你必須嚴格遵循本技能定義的排序規則，不得自行調整排序邏輯。

**範圍限制：本技能僅適用於 RDC 專案。SDC 和 SRN 專案不在範圍內。**

## 排序規則

### 規則 1：Highest 優先順序不得重新排序（鐵律）

所有 `Highest` 優先順序的工單，你不得變更其現有順序或 Rank Number。Highest 工單由專案經理手動管理排序，本技能不介入。

### 規則 2：已有 Rank Number 的工單保留現有順序

對於 `Medium-High`、`Medium`、`Medium-Low`、`Lowest` 優先順序中，已經有 Rank Number（`customfield_10237` 不為空）的工單，你必須保留其現有的相對順序。你不得重新排列這些工單。

### 規則 3：無 Rank Number 的新工單附加至所屬優先順序群組末尾

新工單（`customfield_10237` 為空）必須附加在其所屬優先順序群組中、所有已有 Rank Number 工單的後面。多張新工單之間，按建立日期（created date）由舊到新排序。

### 規則 4：編號區段

每個優先順序群組使用固定的編號起始值：

| 優先順序 | 編號起始值 | 編號範圍 |
|----------|-----------|---------|
| Highest | 手動管理 | 不適用 |
| Medium-High | 201 | 201–299 |
| Medium | 301 | 301–399 |
| Medium-Low | 401 | 401–499 |
| Lowest | 501 | 501–599 |

你必須從該群組的起始值開始，依序遞增 1 進行編號。如果某群組的工單數量超過 99 張，繼續遞增編號（例如 Medium 群組可編到 400 以上），但你必須在輸出中標記警告。

### 規則 5：更新 JIRA Rank Number 欄位

排序完成後，你必須將計算出的 Rank Number 寫入 `customfield_10237` 欄位。每張工單的更新操作使用以下 API 格式：

```
PUT /rest/api/2/issue/{issueKey}
{
  "fields": {
    "customfield_10237": {rankNumber}
  }
}
```

## 排序執行流程

1. 從 JIRA 取得 RDC 專案所有未關閉工單，包含 `priority`、`customfield_10237`、`created` 欄位。
2. 將 Highest 工單排除，不做任何處理。
3. 將剩餘工單按優先順序分為四個群組：Medium-High、Medium、Medium-Low、Lowest。
4. 在每個群組中：
   - 先放置已有 Rank Number 的工單，按現有 Rank Number 由小到大排列。
   - 再放置無 Rank Number 的新工單，按建立日期由舊到新排列。
5. 從該群組的起始編號開始，依序分配新的 Rank Number。
6. 輸出排序結果表格，並列出需要更新的工單清單。

需更新工單數：5 張

## 錯誤處理

- 若某群組工單數量超過 99 張，你必須在輸出中加入以下警告：「警告：{優先順序} 群組工單數量為 {N} 張，已超過預設 99 張上限，編號已延伸至 {最大編號}。」
- 若工單同時屬於 Highest 且有其他優先順序標記（資料異常），你必須以 Highest 為準，不做重新排序，並在輸出中標記該工單為資料異常。
- 若 JIRA API 更新失敗，你必須記錄失敗的工單 Key 與錯誤訊息，並在最終報告中列出所有失敗項目。

## 限制

- 你不得變更 Highest 工單的排序或 Rank Number，無論任何情況。
- 你不得跳過已有 Rank Number 的工單，每次執行必須對該群組全部工單重新編號。
- 你不得對 RDC 以外的專案執行排序操作。
